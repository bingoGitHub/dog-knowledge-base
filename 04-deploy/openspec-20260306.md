# OpenSpec 部署与使用指南

OpenSpec 是一个**规范驱动开发**工具，与 OpenCode 深度集成，帮助开发者通过结构化的规范文件管理项目迭代，特别适合重构、功能演进和团队协作。本文将详细介绍 OpenSpec 的安装、配置及核心工作流程。

---

## 1. 概述

OpenSpec 的核心思想是 **“先写规范，再写代码”**。每个开发任务（变更）都包含提案、设计、任务清单等规范文件，AI 或开发者根据这些规范执行具体编码。变更完成后，规范归档并同步到主规范库，形成项目的“知识库”，确保后续开发有据可依。

---

## 2. 安装

OpenSpec 作为独立的 CLI 工具发布，通过 npm 安装：

bash

npm install -g @fission-ai/openspec@latest

安装完成后，可通过 `openspec --version` 验证。

> **注意**：OpenSpec 与 OpenCode 配合使用效果最佳，但 CLI 命令本身不依赖 OpenCode。

---

## 3. 初始化项目

在项目根目录执行以下命令：

bash

openspec init

初始化过程会：

- 创建 `openspec/` 目录及必要的子目录（`changes/`, `specs/`, `archive/`）
    
- 生成默认配置文件 `openspec/config.yaml`
    
- 生成 `AGENTS.md`（AI 工作指南）
    

初始化完成后，项目结构如下：

text

your-project/
├── openspec/
│   ├── config.yaml          # 全局配置
│   ├── specs/                # 主规范库（初始为空）
│   ├── changes/              # 进行中的变更
│   └── archive/              # 已归档的变更
├── AGENTS.md                  # AI 工作指南（可选）
└── ...

---

## 4. 配置 `config.yaml`

`config.yaml` 是 OpenSpec 的核心配置文件，包含项目上下文和文档生成规则。

### 4.1 项目上下文 (`context`)

`context` 字段为 AI 提供项目背景，每次生成规范时自动注入。建议包含：

- 技术栈（语言、框架、数据库等）
    
- 代码约定（命名风格、测试要求等）
    
- 业务领域（核心概念、术语）
    

示例：

yaml

context: |
  Tech stack: Node.js 18+, TypeScript, Express, Prisma, PostgreSQL
  Architecture: feature-based modular structure
  Code style: ESLint + Prettier, no semicolons, use ES Modules
  Testing: Jest with 80% coverage
  Domain: e-commerce platform (products, orders, inventory)

### 4.2 文档规则 (`rules`)

为不同类型的规范文件（artifact）定义生成规则，确保格式和质量。支持的类型：`proposal`、`tasks`、`design`、`change`。

示例：

yaml

rules:
  proposal:
    - Always include a "Current Problem" section
    - Always include a "Success Criteria" section
  tasks:
    - Break tasks into chunks of max 2 hours
    - Each task must include a verification step
  design:
    - For refactoring, describe "Target Architecture" and "Migration Strategy"

## 5. 目录结构详解

|目录/文件|说明|
|---|---|
|`openspec/config.yaml`|全局配置文件|
|`openspec/specs/`|**主规范库**，存放所有已完成的规范（按模块组织）|
|`openspec/changes/`|**进行中的变更**，每个变更一个子目录，包含 `proposal.md`、`design.md`、`tasks.md` 等|
|`openspec/changes/archive/`|已归档变更的存放位置（按日期命名）|
|`AGENTS.md`|AI 工作指南，可放置项目通用的说明，OpenCode 会自动加载|

---

## 6. 核心工作流程：变更生命周期

OpenSpec 的工作流程围绕“变更”展开，每个变更代表一个独立的任务（新功能、重构、修复等）。流程分为四个阶段：

### 6.1 创建变更 (`/opsx-propose`)

在 OpenCode 中运行：

text

/opsx-propose 为订单模块添加优惠券支持

AI 会自动在 `openspec/changes/` 下创建一个新目录（如 `add-coupon-support`），并生成三个核心文件：

- **`proposal.md`**：描述需求背景、目标、验收标准。
    
- **`design.md`**：技术方案设计（数据结构、API、流程图等）。
    
- **`tasks.md`**：可执行的任务清单（Markdown 复选框）。
    

### 6.2 探索需求 (`/opsx-explore`)

若需求不明确，可先进入探索模式：

text

/opsx-explore 我想重构订单模块，有哪些方案？

AI 会与你讨论，分析利弊。达成共识后，可要求它执行 `/opsx-propose` 生成正式变更。

### 6.3 实施变更 (`/opsx-apply`)

当规范文件准备就绪，运行：

text

/opsx-apply add-coupon-support

AI 会按 `tasks.md` 清单逐项执行（编写代码、运行测试等），每完成一项自动勾选。你可实时查看进度。

### 6.4 归档变更 (`/opsx-archive`)

任务全部完成并验证通过后，归档：

text

/opsx-archive add-coupon-support

归档操作会：

- 将变更目录移至 `openspec/changes/archive/日期-变更名/`
    
- **同步规范到主库**：若本次变更修改了已有规范，则自动合并到 `openspec/specs/` 对应文件中；若为全新功能，则显示 `No main specs to sync`，表示规范已存档但未影响主库。

|命令|作用|

## 8. 规范文件详解

### `proposal.md`（提案）

- 回答 **“为什么做”** 和 **“做什么”**
    
- 包含：背景、目标、验收标准、影响范围
    

### `design.md`（设计）

- 回答 **“怎么做”**
    
- 包含：技术方案、数据模型、API 设计、迁移策略
    

### `tasks.md`（任务清单）

- 可执行步骤，使用 Markdown 复选框 `- [ ]`
    
- 每个任务应具体、可验证（如“添加单元测试”、“运行 lint”）

## 9. 最佳实践

### 9.1 在现有项目中引入 OpenSpec

1. 运行 `openspec init` 初始化。
    
2. 完善 `config.yaml` 中的 `context` 和 `rules`。
    
3. 从当前需要做的第一个任务开始，用 `/opsx-propose` 创建变更。
    
4. 随着多个变更归档，`specs/` 目录逐渐丰富，形成项目知识库。
    

### 9.2 为重构任务创建变更

- 在 `proposal.md` 中明确“当前问题”和“成功标准”。
    
- 在 `design.md` 中描述目标架构和迁移步骤。
    
- 在 `tasks.md` 中拆分细粒度任务，并包含验证步骤（如运行测试）。
    

### 9.3 手动填充主规范库

若希望将已有文档纳入 `specs/`，可手动复制归档变更中的规范文件：

bash

cp -r openspec/changes/archive/2026-03-05-split-call-under/* openspec/specs/split-call-under/

### 9.4 配置建议

- 在 `config.yaml` 的 `context` 中写明项目核心信息，让 AI 生成的规范更准确。
    
- 为 `tasks.md` 添加规则，确保任务可验证（如 `Each task must include a verification step`）。

## 10. 常见问题

### Q: 归档时显示 “No main specs to sync” 是什么意思？

A: 表示本次变更没有影响已有的主规范（例如第一次归档，或新增模块）。这是正常现象，规范文件已存档，后续修改该模块时会自动同步。

### Q: 如何让 AI 参考已归档的规范？

A: 下次创建涉及同一模块的变更时，AI 会自动读取 `specs/` 中已有的规范。也可在对话中明确引用，如 `@openspec/specs/order/design.md`。

### Q: 能否在 OpenCode 之外使用 OpenSpec？

A: 可以。OpenSpec CLI 提供了 `openspec change new`、`openspec archive` 等命令，但配合 OpenCode 可获得 AI 驱动的自动化体验。
